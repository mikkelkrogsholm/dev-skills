# Making Code Easier for AI Coding Agents to Read and Modify

Recommendations for structuring code, repos, and conventions so that AI coding agents (Claude Code, Cursor, Copilot, Aider) make fewer mistakes. Each recommendation is grounded in a specific failure mode observed in practice.

Prioritization legend:
- **P0** — High leverage, low cost. Do these first.
- **P1** — Significant impact, modest effort.
- **P2** — Worthwhile polish once P0/P1 are done.

---

## P0 — Structural foundations

### 1. Provide an agent-readable project map (`CLAUDE.md` / `.cursorrules` / `AGENTS.md`)

**Failure mode:** Agents open random files, misidentify where logic lives, and reinvent existing utilities. Default behavior is to grep the repo from scratch every session, often missing patterns that exist in adjacent modules.

**Do:**
- At the repo root, keep a short file (≤150 lines) that states: the stack, folder layout with one-line purpose per directory, the commands to run/test/lint, and hard rules ("never import X from Y", "all DB writes go through `db/writer.ts`").
- List where *not* to look (generated folders, vendored code) explicitly. Agents otherwise waste tokens there.

**Example:**
```md
## Layout
- `src/api/` — HTTP handlers. One file per route group.
- `src/domain/` — Pure business logic. No I/O, no imports from `api/` or `db/`.
- `src/db/` — Drizzle schema + queries. Never import from `domain/`.
- `src/generated/` — DO NOT EDIT. Regenerated by `bun run codegen`.

## Commands
- Test: `bun test`
- Typecheck: `bun run tsc --noEmit`
- Lint: `bun run lint --fix`
```

### 2. Make the type system the source of truth

**Failure mode:** Agents infer types from variable names and usage. When types are implicit (`any`, untyped JSON, stringly-typed enums), they generate plausible-but-wrong code — e.g., assuming `user.role` is `"admin" | "user"` when it's actually free-form.

**Do:**
- Turn on `"strict": true` in `tsconfig.json`. Add `noUncheckedIndexedAccess` and `exactOptionalPropertyTypes`.
- Validate all external inputs (HTTP bodies, env vars, DB rows from raw SQL) with Zod/Valibot and export the inferred type.
- Replace string literal unions that appear in >1 file with a named type.

**Counter-example (bad):**
```ts
function sendEmail(kind: string, user: any) { /* ... */ }
```
The agent will invent new `kind` values in callers.

**Good:**
```ts
type EmailKind = "welcome" | "reset-password" | "receipt";
function sendEmail(kind: EmailKind, user: User) { /* ... */ }
```

### 3. Co-locate related code; avoid deep abstraction ladders

**Failure mode:** When logic is split across `BaseController → AbstractController → UserController → UserService → UserRepository → UserDao`, agents either duplicate logic at the wrong level or edit the wrong layer. They cannot hold the full call chain in context.

**Do:**
- Prefer a flat file with a complete feature over a spread across 5 files.
- Inline single-use helpers. Abstract only when there are ≥3 call sites with genuinely shared behavior.
- Keep a feature's route, validation, handler, and types within 1–2 files when possible.

### 4. Name files after what they contain, not what they are

**Failure mode:** Agents grep for `createUser` and find it in `utils.ts`, `helpers.ts`, and `index.ts`. They edit the wrong one.

**Do:**
- Avoid `utils.ts`, `helpers.ts`, `common.ts`, `misc.ts`. Name by domain: `password-hashing.ts`, `date-formatting.ts`.
- One primary export per file. Filename matches the export.
- Avoid barrel `index.ts` re-exports across the whole app — they make "go to definition" unreliable and hide where code lives.

## P1 — Code-level signals

### 5. Write tests that double as specs

**Failure mode:** Agents modify a function and break an invariant that was never documented. They don't know the intended semantics and ship code that typechecks but violates a business rule (e.g., "invoice totals must include tax but exclude shipping discounts").

**Do:**
- Every non-trivial function should have at least one test describing its contract in the `it(...)` string.
- Prefer `describe("calculateInvoiceTotal", () => { it("includes tax but excludes shipping discounts", ...) })` over `it("works")`.
- Add a test for every bug fix; the test name states the invariant.

**Why it helps:** Agents read test names as executable docs. A failing test with a clear name tells them *why* their change is wrong.

### 6. Make side effects explicit and localized

**Failure mode:** A function named `getUser` also writes an audit log and refreshes a cache. An agent calls it in a loop or in a pure context and causes N+1 writes, cache stampedes, or test pollution.

**Do:**
- Pure functions should have no I/O. Name impure functions with verbs that imply action: `persistUser`, `fetchAndCacheUser`.
- Keep DB writes, HTTP calls, and file I/O at the edges (handlers, workers). Domain code takes data in, returns data out.
- Use a "ports and adapters" split even informally: the domain function takes a `repo` parameter instead of importing the DB directly.

### 7. Prefer exhaustive switches over default cases

**Failure mode:** When a new variant is added to a union, agents forget to update every `switch`/`if` chain. Silent fallthrough via `default` hides the miss.

**Do:**
```ts
function label(k: EmailKind): string {
  switch (k) {
    case "welcome": return "Welcome";
    case "reset-password": return "Reset your password";
    case "receipt": return "Your receipt";
    default: {
      const _exhaustive: never = k;
      throw new Error(`Unhandled kind: ${_exhaustive}`);
    }
  }
}
```
TypeScript's `never` check makes the agent's missing case a compile error instead of a runtime one.

### 8. Keep functions short and single-purpose, but do not over-split

**Failure mode:** 300-line functions cause agents to lose track of state and introduce subtle bugs near the middle. Conversely, functions that are one-line wrappers around another function force agents to chase references.

**Do:**
- Aim for 5–40 lines per function. Break up when scrolling required.
- Don't extract until there's a second caller or the block has a clear name. Premature extraction produces `processUserData` / `handleUserData` / `doUserData` that agents confuse.

### 9. Explicit imports over wildcards and dynamic resolution

**Failure mode:** `import * as utils` or runtime `require(name)` breaks the agent's ability to find where a symbol is defined. Agents will fabricate a function that doesn't exist because they can't resolve the import.

**Do:**
- Named imports only: `import { hashPassword } from "./password-hashing"`.
- No `eval`, no `require()` with computed names, no `import()` with variables in application code.
- No TypeScript path aliases that shadow package names. `@/utils` is fine; `react` as a local alias is not.

### 10. Keep configuration colocated with the code it configures

**Failure mode:** Agent edits `src/email/send.ts` but the retry limit lives in `config/constants.ts` and the timeout in `.env.example`. The agent misses one, produces an inconsistent change.

**Do:**
- Put small config next to its consumer (`src/email/config.ts`).
- If you have a central config, make it a typed module that re-exports from per-feature files, not a 500-line dump.
- `.env.example` must list every env var the app reads. Agents treat its absence as "this var doesn't exist".

## P1 — Repository hygiene

### 11. Commit lockfiles, commit generated schemas, exclude build output

**Failure mode:** Agents run `npm install` and pull a newer incompatible version. Or they regenerate a Prisma client that's out of sync with the DB.

**Do:**
- Commit `bun.lockb` / `pnpm-lock.yaml` / `package-lock.json`.
- Commit generated artifacts that are read by humans/agents (OpenAPI specs, GraphQL schemas, Prisma client types when relevant).
- Gitignore build output (`dist/`, `.next/`). Agents that see both source and build output will sometimes edit build output.

### 12. One command to validate a change

**Failure mode:** Agents ship code that compiles but fails lint, or passes lint but fails tests, because they only ran one check.

**Do:**
- Provide `bun run check` (or equivalent) that runs typecheck + lint + test in sequence.
- Document it in CLAUDE.md. Agents will run what's documented.

### 13. Deterministic formatting and linting with autofix

**Failure mode:** Agent diff contains 80% whitespace changes because their editor reformats on save. Review becomes impossible.

**Do:**
- Commit Prettier/Biome config. Enforce via pre-commit hook or CI.
- Prefer Biome (single tool) or Prettier + ESLint with `eslint-config-prettier` to avoid fights.
- Turn on `--fix` in the `check` command so the agent's first run produces a clean diff.

## P1 — Documentation that agents actually use

### 14. Inline comments for *why*, not *what*

**Failure mode:** Agents delete "redundant" checks (`// check if user is null`) that were actually guarding against a known edge case. Or they duplicate a workaround because the original had no comment explaining it.

**Do:**
- Comment non-obvious constraints: `// Stripe webhooks can retry within 5 minutes; we dedupe on event.id`.
- Link to issues/PRs/docs for anything surprising: `// Workaround for https://github.com/org/repo/issues/123`.
- Don't comment what the code plainly says.

### 15. Decision records for architectural constraints

**Failure mode:** Agent "cleans up" code by removing a layer, not realizing it existed to satisfy a constraint (multi-tenancy, GDPR data residency, a vendor API quirk).

**Do:**
- A simple `docs/adr/` folder with short (1 page) decision records. Title like `0007-why-we-dont-use-orm-relations-for-tenant-isolation.md`.
- Reference ADRs from code comments at the relevant call sites: `// See ADR-0007`.

### 16. JSDoc on public APIs — especially for non-obvious parameters

**Failure mode:** A function takes `timeout: number` and the agent passes milliseconds when it wanted seconds.

**Do:**
```ts
/**
 * @param timeoutMs - upper bound in milliseconds. Server hard-caps at 30_000.
 * @param signal - abort propagates to underlying fetch; on abort the job row is marked 'cancelled'.
 */
```
Units in parameter names (`timeoutMs`, `sizeBytes`, `priceCents`) reduce ambiguity even without JSDoc.

## P2 — Advanced hygiene

### 17. Avoid "magic" metaprogramming

**Failure mode:** Decorators that rewrite behavior, Proxy-based ORMs that expose methods not in the type definitions, Vite/Webpack plugins that transform imports — agents see code that doesn't exist in the file. They "fix" what they perceive as bugs.

**Do:**
- Prefer explicit function calls over decorators where possible.
- If you use a framework with heavy magic (NestJS, tRPC router merging, etc.), put a "how this works" section in CLAUDE.md with pointers.

### 18. Keep error handling uniform

**Failure mode:** One handler throws, another returns `{ok: false, error}`, a third calls `res.status(500).json(...)`. Agent copies a handler and uses the wrong convention.

**Do:**
- Pick one: exceptions, Result types, or error responses — and stick to it per layer.
- Document the convention in CLAUDE.md. Example: "Domain functions throw `DomainError`. API handlers convert to HTTP responses via `errorMiddleware`."

### 19. Stable test data and fixtures

**Failure mode:** Agent writes a test using random faker data; it passes locally, flakes in CI. Or it hardcodes IDs that conflict with existing fixtures.

**Do:**
- Fixture factories with sensible defaults: `makeUser({ role: "admin" })`.
- Seeded randomness (`faker.seed(1)`) if you use faker at all.
- Never use `Date.now()` in tests — inject a clock.

### 20. Log and error messages that are greppable

**Failure mode:** Agent debugs by searching the log message in code and can't find it because it's interpolated: `` `Failed to process ${kind}` `` matched as "Failed to process".

**Do:**
- Prefix log messages with a stable tag: `[email:send] timeout after 30s`.
- Keep a static leading string so grep finds the source.

### 21. Prefer composition over inheritance; avoid mixins

**Failure mode:** Agents cannot reliably resolve `this.doThing()` when `doThing` is provided by a mixin three levels up. They either call the wrong override or duplicate the method.

**Do:**
- Plain functions + plain data. Classes for genuine stateful objects only.
- No inheritance past one level in application code.

### 22. Make failure states first-class, not exceptional

**Failure mode:** Code treats happy path as the shape (`user.email`) and errors as thrown exceptions, so agents forget to handle `null`/failures when refactoring.

**Do:**
- For operations that can fail predictably, return a discriminated union: `{ ok: true, data } | { ok: false, error }`. Agents see both cases in the type and handle both.
- Reserve exceptions for truly unexpected conditions.

## Anti-patterns to remove

- `any`, `@ts-ignore`, `// eslint-disable-next-line` without justification comment
- `TODO` without a ticket link or date — agents will "resolve" them incorrectly
- Commented-out code left in the repo — agents resurrect it
- Multiple conflicting READMEs (`README.md`, `README.old.md`, `docs/README.md`)
- Secrets in `.env` that isn't gitignored, or `.env.example` that lies about what vars exist
- Deep relative imports (`../../../../config`) — switch to path aliases
- Files named by date or author (`user_v2.ts`, `user_new.ts`, `user_john.ts`)

---

## Quick checklist

Before declaring a codebase "agent-friendly":

- [ ] CLAUDE.md (or equivalent) exists, <150 lines, states layout + commands + hard rules
- [ ] `strict: true` TypeScript, external inputs validated
- [ ] One command validates everything (`check`)
- [ ] Tests describe invariants in their names
- [ ] No files named `utils.ts` / `helpers.ts` / `common.ts`
- [ ] Exhaustive switches on unions
- [ ] `.env.example` complete
- [ ] Lockfile committed, build output gitignored
- [ ] Comments explain *why*, ADRs exist for cross-cutting decisions
- [ ] One error-handling convention per layer, documented
- [ ] No magic metaprogramming, or it's documented in CLAUDE.md

The payoff: agents spend tokens changing code, not re-deriving context. First-try diffs succeed more often. Review surface shrinks.
